  AI ASSISTANT CHAT HISTORY SYSTEM - COMPREHENSIVE TASK LIST
═══════════════════════════════════════════════════════════════════════════════

PROJECT OVERVIEW:
Build a complete chat history system for the AI Assistant that allows users to:
- Save and manage multiple conversation threads
- View conversation history in a modern, sleek modal interface
- Continue previous conversations seamlessly
- Search and filter conversations
- Delete or archive old conversations

PHASE 1: DATABASE DESIGN & SETUP
═══════════════════════════════════════════════════════════════════════════════

[ ] 1.1 Design Supabase SQL Schema
    - Create 'ai_conversations' table
      * id (uuid, primary key)
      * user_id (uuid, foreign key to auth.users)
      * title (text) - Auto-generated from first message or user-defined
      * created_at (timestamp with time zone)
      * updated_at (timestamp with time zone)
      * is_archived (boolean, default false)
      * message_count (integer, default 0)
      * preview_text (text) - First 100 chars of conversation
      * metadata (jsonb) - For extensibility
    
    - Create 'ai_messages' table
      * id (uuid, primary key)
      * conversation_id (uuid, foreign key to ai_conversations)
      * role (text) - 'user' or 'assistant'
      * content (text)
      * created_at (timestamp with time zone)
      * tokens_used (integer, nullable)
      * metadata (jsonb) - For tool calls, errors, etc.

[ ] 1.2 Set up RLS (Row Level Security) Policies
    - ai_conversations: Users can only access their own conversations
    - ai_messages: Users can only access messages from their conversations
    - Service role admin access for both tables

[ ] 1.3 Create Database Indexes
    - Index on ai_conversations.user_id for fast user lookups
    - Index on ai_conversations.updated_at for sorting
    - Index on ai_messages.conversation_id for message retrieval
    - Index on ai_messages.created_at for chronological ordering

[ ] 1.4 Create Database Functions
    - update_conversation_timestamp() - Trigger to update updated_at
    - increment_message_count() - Trigger to update message_count
    - update_preview_text() - Function to update conversation preview

PHASE 2: BACKEND API DEVELOPMENT
═══════════════════════════════════════════════════════════════════════════════

[ ] 2.1 Create API Route: /api/ai-assistant/conversations
    - GET: List all conversations for authenticated user
      * Support pagination (limit, offset)
      * Support sorting (created_at, updated_at)
      * Support filtering (archived, search query)
      * Return: { conversations: [], total: number, hasMore: boolean }
    
    - POST: Create new conversation
      * Input: { title?: string, firstMessage?: string }
      * Auto-generate title if not provided
      * Return: { conversation: {...}, id: string }

[ ] 2.2 Create API Route: /api/ai-assistant/conversations/[id]
    - GET: Retrieve full conversation with all messages
      * Include pagination for messages if conversation is very long
      * Return: { conversation: {...}, messages: [] }
    
    - PATCH: Update conversation
      * Allow updating: title, is_archived
      * Return: { conversation: {...} }
    
    - DELETE: Delete conversation and all messages
      * Cascade delete all associated messages
      * Return: { success: boolean }

[ ] 2.3 Create API Route: /api/ai-assistant/conversations/[id]/messages
    - POST: Add message to conversation
      * Input: { role: 'user' | 'assistant', content: string, metadata?: {} }
      * Update conversation.updated_at and message_count
      * Update preview_text if first message
      * Return: { message: {...} }

[ ] 2.4 Update Existing Chat API
    - Modify /api/ai-assistant/chat to accept conversation_id
    - Save all messages to database when conversation_id provided
    - Create new conversation if conversation_id not provided
    - Return conversation_id in response
